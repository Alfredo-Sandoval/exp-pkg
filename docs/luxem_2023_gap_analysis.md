# Response to Luxem et al. 2023

Luxem K, Sun JJ, Bradley SP, Krishnan K, Yttri E, Zimmermann J, Pereira TD,
Laubach M. *Open-source tools for behavioral video analysis: Setup, methods,
and best practices.* eLife. 2023;12:e79305.
[doi:10.7554/eLife.79305](https://doi.org/10.7554/eLife.79305)

## Scope

The paper describes a fragmented behavioral-video workflow spanning
acquisition, calibration, pose estimation, multi-animal identity, behavior
quantification, synchronization, and FAIR sharing. `exp-pkg` responds by
providing a common typed IO boundary for those products. It does not train pose
models, discover behavior motifs, or replace analysis packages.

Raw files and vendor exports enter through readers and importers. They are
converted once into the ontology described in
[Experiment Ontology](architecture/experiment-ontology.md). Interior project
code operates on those objects rather than format-specific dictionaries.

## Supported experiment shapes

The ontology can represent:

- one experiment with one or many recording sessions
- repeated or longitudinal sessions involving the same subjects
- one or many biological subjects in a session
- conditions applied to a session, cohort, or selected participating subjects
- distinct experimental and acquisition protocols
- one or many cameras with explicit acquisition-camera links
- single-camera or multi-camera calibration
- 2D image-space pose and 3D camera, calibration-world, marker-world, or
  model-lifted pose
- one or many technical pose tracks, with explicit subject-to-track identity
  assignments
- pose products generated by different tools or checkpoints in different
  sessions
- videos, events, behavior labels, sampled signals, and photometry
- multiple timebases with evidence-backed alignment records
- dataset sharing metadata, datasheets, model cards, and portable project
  artifacts

## Canonical response

The storage project is not a scientific session. It is the transactional and
portable container for one `Experiment` aggregate. The experiment owns
registries of `Subject`, `Protocol`, and `ExperimentalCondition` objects and
links them to any number of `RecordingSession` objects.

Each recording session owns acquisition context and typed modality links.
Videos link directly to acquisition cameras. Calibrations link acquisition
cameras to calibrated camera models. Pose links identify their videos,
calibration, coordinate frame, tracks, and model provenance. A
`SubjectTrackLink` connects biological identity to a technical pose track
without treating a tracker-assigned ID as the animal itself.

`PoseTrajectory` uses the shape `(frames, tracks, keypoints, dimensions)`.
The explicit track axis is required for multi-animal 2D and 3D results.
`Labels` remains the annotation-rich 2D representation. Both are valid pose
products on a session.

## Current IO coverage

- DeepLabCut CSV, H5, and project imports
- SLEAP analysis H5 and package imports
- Lightning Pose CSV, MMPose JSON, and MediaPipe JSON imports
- Anipose TOML and OpenCV stereo YAML calibration imports
- generic and vendor-specific photometry readers
- generic events and behavior readers, including BORIS, SimBA, B-SOiD, and
  Keypoint-MoSeq exports
- typed 2D and 3D pose storage
- typed acquisition, camera, calibration, timing, event, signal, behavior,
  subject, protocol, and condition objects
- project validation, shallow inspection, and `.expkg` packaging

## Remaining IO work

The ontology is broad enough for the experiment shapes above, but format
coverage remains incomplete. The next additions should be boundary adapters
and evidence-based inspection, not parallel models:

1. Add 3D result importers for common Anipose, DANNCE, DeepFly3D, and related
   exports.
2. Add project actions and importers for event, behavior, and synchronization
   files.
3. Add Kalibr and MC-Calib calibration importers.
4. Add shallow acquisition and synchronization QC only when the source records
   the required evidence.
5. Extend subject-to-track import where source formats contain verified
   biological identity assignments.

## Rejected designs

- **Project equals session.** Rejected because longitudinal and repeated
  experiments require multiple sessions in one scientific aggregate.
- **Acquisition metadata at project scope.** Rejected because cameras, arenas,
  lighting, clocks, and settings can change between sessions.
- **One project-wide pose-provenance sidecar.** Rejected because sessions and
  pose products can use different tools, models, and checkpoints.
- **Calibration JSON outside the session.** Rejected because calibration is a
  session modality with explicit camera relationships.
- **Separate 2D and 3D experiment ontologies.** Rejected because coordinate
  dimensionality and coordinate frame are properties of a pose product.
- **Tracker identity equals subject identity.** Rejected because technical
  tracks can swap, split, merge, or remain anonymous.
- **Tool-specific assay subclasses.** Rejected because they would reproduce the
  fragmentation the package is intended to remove. Protocols, conditions,
  events, and typed modality links carry the scientific meaning without binding
  the ontology to one analysis package.

## Boundary

Pose training, behavior-classifier training, unsupervised motif discovery,
ReID algorithms, and model zoos remain outside this repository. `exp-pkg`
stores their inputs and outputs with enough context and provenance for another
tool to consume them reliably.
